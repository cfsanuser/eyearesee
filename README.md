1. Core Architecture and Setup
Framework: Asynchronous programming (asyncio) for concurrent network operations (IRC connection, AI inference, TUI rendering).
Interface: Curses (curses) for creating a full-screen, interactive terminal user interface (TUI).
Modularity: Supports multiple AI providers (Anthropic, OpenAI) and local/offline LLMs (Ollama, llama.cpp).

Key Setup Functions:
main(): Handles initial configuration: prompting the user for server address, nickname, channel, and NickServ password.
_ensure_deps(): Checks for required Python packages (anthropic, openai, transformers, torch) and automatically installs them via pip if missing, ensuring the AI detection features are available.
main_curses(): Initializes the curses environment, sets up color pairs based on predefined themes, and starts the main asynchronous loop.
2. AI Integration and Detection Engine
This is the most complex feature, responsible for classifying chat messages as human-written or AI-generated.

A. AI Model Registry (AI_MODELS)
A central dictionary defining all available models and their providers:

Cloud Models: opus (Claude Opus), sonnet (Claude Sonnet), haiku (Claude Haiku), gpt4o, gpt4, gpt35 (OpenAI).
Local/Offline Models: gemma (Ollama), llama3 (Ollama).
Local/Offline Models (llama.cpp): gemma4.
B. AI Detector (EnsembleAIDetector)
This class uses an ensemble of ML models and heuristic signals to determine the probability that a message was generated by AI.

Primary Classifier (_cls_model): A RoBERTa model trained on ChatGPT/GPT-4 output, strong for detecting high-quality, fluent AI text.
Secondary Classifier (_cls2_model): A RoBERTa model trained on GPT-2 output, generalized to detect broad fluency patterns common across various open-source LLMs (Llama, Mistral).
Heuristic Scoring: Calculates scores based on several linguistic features:
formality_score: Measures classic IRC vs. formal/LLM language usage.
llama_pattern_score: Detects structural signals like markdown (lists, headers), bot-opener phrases, and long, uniform sentences typical of LLM outputs.
repetition & entropy: Measures linguistic predictability and complexity.
Binoculars Score (_binoculars_score): Uses GPT-2 family models to measure the agreement between two different models (observer vs. performer). Low agreement suggests high fluency (likely AI).
Ensemble Probability (predict_detailed): Combines the results of the ML signals (Binoculars, Classifiers, Heuristics) into a final probability score (0.0 to 1.0).
LLM-Specific Classification (_llm_classify_ai): When a specific model is selected (via /model), this function queries the chosen provider (Anthropic, OpenAI, Ollama, llama.cpp) to get a direct AI/Human classification.
3. Data Logging and Historical Analysis
The system maintains two types of logs: real-time event logs and persistent AI scoring logs.

AI Score Logging (ai_scores.log - JSONL format):
Records every message event, including user score, message score, ensemble AI score, heuristic breakdown (formality, llama pattern, classifier scores), and message length.
Gap Detection: Tracks sequential message numbers (seq) per session to identify missing or injected messages.
History Loading:
load_nick_history(nick): Parses the AI log to aggregate all scores, message lengths, and session data for a specific user, providing statistics like total messages, first/last activity, and a list of top-scored messages.
load_historical_suspects(threshold): Analyzes the entire log to identify users with a consistently high average AI score, providing a long-term risk assessment.
Log Toggling: (/logtoggle) allows the user to enable or disable the persistent AI scoring log file, with an auditable "session_start" and "log_toggle" record for tracking changes.
4. IRC Client and Network Handling (IRCClient)
The client is built around robust asynchronous networking and flood control.

Connection Management (connect, run_connection): Handles establishing connections (plain or SSL) and implements exponential backoff for automatic reconnection upon failure.
Flood Control (_run_writer): Implements a Token Bucket algorithm to limit the sending rate to 4 lines per second with a burst capacity of 10 lines. This prevents the client from being kicked by IRC servers due to flooding (e.g., mass JOINs or MOTD bursts).
CTCP Handling (cmd_ctcp): Manages CTCP (Client-to-Client Protocol) requests, which are used for server status (PING, TIME, CLIENTINFO). It implements a rate limiter (_ctcp_allowed) to restrict the number of CTCP replies per nick per 30 seconds.
Command Dispatch (_build_irc_handlers): Maps standard IRC commands (PRIVMSG, JOIN, KICK, MODE, etc.) to asynchronous handler functions.
Message Handling (process_line): Parses raw IRC lines, extracts commands, nicks, parameters, and IRCv3 message tags.
5. Terminal User Interface (TUI)
The TUI provides a multi-pane, interactive dashboard for managing the connection, viewing chat, and accessing AI features.

A. Window Management
Multi-Server Support: Can manage multiple concurrent IRC connections, each with its own state context (ServerContext).
Window Types: Supports status windows (*status*), chat windows (channels or DMs), and a dashboard window (*dashboard*).
Tab Bar: Provides a navigable strip at the top of the input pane, allowing quick switching between active connections and windows.
B. Drawing and Formatting
IRC Formatting: Functions (irc_parse_formatting, _irc_visual_pos, _str_visual_width) correctly handle IRC control codes (color, bold, italic, reverse) and the complex rendering of Wide/CJK characters (which occupy two terminal columns) to ensure accurate positioning and truncation.
Chat Rendering (_draw_chat): Renders messages, applying formatting and handling line wrapping to fit the terminal width.
User List Rendering (_draw_userlist): Displays users in the current channel, visually marking users whose AI score exceeds the configured threshold (AI_SUSPECT_THRESHOLD) with a bold color.
C. Input and Commands
Input Editing: Implements standard terminal editing features (cursor movement, backspace, Ctrl+A/E/K/U/W) and specialized completion logic (do_nick_complete) for auto-suggesting user names.
Slash Commands (_slash_...): A comprehensive command palette for all major functions:
Chat Control: /msg, /join, /part, /topic, /kick, /invite.
User Info: /nick, /whois, /whowas, /ignore, /away.
AI Control: /model (set/list AI models), /askai (query AI), /summarize (summarize recent chat), /topai (rank users by AI score), /aitoggle, /logtoggle.
Connection Control: /server (add parallel server), /reconnect, /theme.
Interface Control: /win (switch windows), /close, /clear.
D. Dynamic Updates
Dashboard Auto-refresh: The dashboard automatically rebuilds the suspect list every 1 second (or on user navigation) to reflect the current session's AI scores.
Profile Views: Commands like /ai and /summarize lock the dashboard into a "profile" view for 60 seconds to allow the user to read the full output without interruption.
Translation: If enabled (/autotranslate), CJK messages are asynchronously translated to English using Google Translate, with results appended as indented lines in the chat window.

The AI-based commands in the script are designed to give the user deep, actionable insights into the behavior of other users, the performance of the AI detection system, and the capabilities of the integrated LLMs.

These commands are centralized under the /ai umbrella, which primarily focuses on individual user profiles, but it interacts heavily with other commands like /model, /askai, and /summarize.

Here is a detailed breakdown of the /ai related commands and their functions:

1. /ai (User Profile Analysis)
The primary function of /ai is to provide a comprehensive, deep dive into the AI behavior profile of a specific IRC user.

Functionality:
Input: Requires a target nick (/ai <nick>).
Data Aggregation: It concurrently gathers two types of data:
In-Session Data: Real-time metrics (rolling AI likelihood, peak/low scores, message burst rate, message length uniformity, current channel presence).
Historical Data: It asynchronously queries the persistent ai_scores.log to retrieve all past messages, scores, and session data for that user.
Verdict Generation: It synthesizes the gathered data to generate a final verdict, classifying the user's pattern as:
HIGH RISK: Persistent, consistent AI pattern across multiple sessions.
SUSPECT: Elevated AI score.
MODERATE: Borderline pattern.
LOW: No strong AI signal.
Visualization: The results are rendered in the dedicated Dashboard window (*dashboard*), showing:
Session-specific metrics (e.g., message count, avg message length, message frequency).
Trend analysis (e.g., recent score changes).
Historical statistics (all-time averages, peak scores, session history).
Visual indicators (sparklines) showing score volatility.
In essence, /ai <nick> is the ultimate investigative tool, allowing the user to determine if a user is consistently generating AI-like text.

2. Related AI Commands (The AI Ecosystem)
The /ai command works in concert with other commands to control and interact with the AI features.

A. /model [key] (AI Model Selection)
Purpose: To set which AI model the client should use for general tasks.
Action: Allows the user to select from the registered models (e.g., opus, gpt4o, gemma, llama3).
Dual Role: This command serves two roles:
Chat Model: Sets the model used for the /askai and /summarize commands.
Detection Model: Sets the model used by the EnsembleAIDetector for real-time message scoring.
B. /askai [model] <question> (LLM Query)
Purpose: To ask a specific AI model a question and display the answer.
Action:
It selects the specified model (or the currently set default) to use.
It sends the question to the chosen provider (Anthropic, OpenAI, Ollama, etc.).
The resulting answer, along with the token usage, is displayed in the Dashboard window.
C. /summarize [n] [model] (Chat Transcript Analysis)
Purpose: To summarize the recent chat history in the current window using an AI model.
Action:
It extracts the last n messages from the active chat window.
It constructs a detailed prompt for the AI, instructing it to perform a structured analysis (topics, user contributions, interactions, conclusions).
It queries the chosen AI model with the transcript.
The resulting structured analysis is displayed in the Dashboard window.
D. /topai (Channel Ranking)
Purpose: To rank the users in the current channel based on their current AI scores.
Action:
It collects the rolling AI likelihood score for every user currently in the channel.
It sorts the users by their AI score in descending order.
It displays a ranked list in the Dashboard window, highlighting users who meet or exceed the AI_SUSPECT_THRESHOLD.
